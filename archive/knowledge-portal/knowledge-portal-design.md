# 知识门户对接设计

## 1. 文档目的

本文用于统一收敛知识门户相关的当前结论，覆盖以下内容：

- 门户首页模块拆分
- 门户与 BiSheng 知识空间的数据边界
- 门户 API 的最小设计方向

本文是当前知识门户对接的总设计文档。

统一内容 schema 不再放在本文中，单独维护在：

- `docs/architecture/knowledge-portal-unified-schema.md`
- `docs/architecture/knowledge-portal-api-spec.md`

## 2. 调研范围与前提

调研时间：`2026-04-13`

当前对齐的门户目标站点：

- 首页：`https://kg.nxuhe.com/`
- 技术问答页：`https://kg.nxuhe.com/portal/qa`
- 知识详情页：`https://kg.nxuhe.com/portal/k/rec-hotroll-1580-vibration`

当前前提约束：

- BiSheng 的知识空间只提供数据
- 门户前台负责页面组织、搜索体验、推荐编排、问答体验与分享流程
- 门户内容默认完全公开
- 首页不做独立的 `home-data` 聚合接口
- 首页采用“前端定规则，后端给结果”的模式

## 3. 当前站点结论

当前站点不是纯静态页，也不是已经完整接通 BiSheng 的正式门户，而是：

- 有真实路由的 Next.js 门户前台
- 部分页面已有明确内容模型
- 大量交互仍处于演示或占位状态

已确认存在独立页面的能力：

- 门户首页
- 技术问答页
- 知识详情页

已确认仍为演示或占位的能力：

- 首页部分“更多”“知识广场”“应用市场”点击逻辑仍为提示或占位
- 技术问答页回复为模拟内容，会话历史仅存在前端页内存
- 问答页的“联网搜索”“文件搜索”“语音输入”等能力尚未接真实后端

## 4. 首页模块拆分

### 4.1 顶部导航

包含内容：

- 品牌 Logo
- 门户名称
- 登录/用户入口
- 顶部导航入口

结论：

- 前端写死
- 登录/用户入口直接复用 BiSheng 登录体系
- 不需要单独 API

### 4.2 全局搜索区

包含内容：

- 搜索框
- 快捷搜索词
- 搜索按钮
- 搜索提示/建议

结论：

- 这是门户首页的全局搜索入口
- 搜索产品能力属于门户
- 搜索结果依赖后端接口返回

### 4.3 焦点展示区

包含内容：

- Banner 轮播
- 重点专题入口
- 首屏主视觉文案

结论：

- 当前由前端写死
- 不依赖 API

### 4.4 业务域入口区

包含内容：

- 业务域卡片/图标
- 业务域名称
- 进入业务域内容集合的入口

结论：

- 分类壳层前端写死
- 各业务域对应的内容由后端返回

### 4.5 推荐内容区

包含内容：

- 推荐知识
- 最新内容
- 重点内容
- 首页主推荐列表

结论：

- 推荐区域在前端配置
- 前端手动绑定推荐区域对应的知识空间
- 后端只根据前端传入的 `space_id` 返回该知识空间下的文档列表

### 4.6 专题内容区

包含内容：

- 案例
- 制度
- 课程
- 词条
- 其他专题栏目

结论：

- 专题区域在前端配置
- 前端手动绑定专题区域对应的知识空间
- 后端只根据前端传入的 `space_id` 返回该知识空间下的文档列表

### 4.7 知识广场区

包含内容：

- 知识空间/知识库卡片
- 知识库简介
- 知识库入口
- 展开全部

结论：

- 前端控制分类和对应知识空间
- 前端手动配置和绑定相关入口或功能
- 知识空间卡片对象本身由前端写死配置
- 后端只按 `space_id` 返回对应知识空间下的内容列表

### 4.8 技术问答入口区

包含内容：

- 热门问题
- 去提问入口
- 问答能力说明

结论：

- 首页只保留问答入口
- 实际能力通过 BiSheng 日常模式 API 实现
- 参与问答的知识库范围由前端手动选择和配置
- 不做首页专属问答 API

### 4.9 应用市场区

包含内容：

- 智能体卡片
- 工具/应用入口
- 应用说明

结论：

- 前端直接配置和绑定 BiSheng Workflow 的免登录 URL
- 不走门户 API
- 用户点击后直接跳转对应 URL
- 不需要单独 API

### 4.10 页脚信息区

包含内容：

- 门户说明
- 版权信息
- 辅助链接

结论：

- 前端写死
- 不依赖 API

## 5. 首页模块分组

### 5.1 前端写死，不需要 API

- 顶部导航
- 焦点展示区
- 应用市场区
- 页脚信息区

### 5.2 前端控制规则，后端返回结果

- 全局搜索区
- 业务域入口区
- 推荐内容区
- 专题内容区
- 知识广场区

### 5.3 依赖现有能力，不新增首页专属 API

- 技术问答入口区

## 6. 门户与知识空间的数据边界

### 6.1 门户层负责什么

以下能力归属于门户前台，不建议直接压到知识空间内实现：

- 首页、顶部导航、品牌区、登录/用户菜单
- 搜索框、快捷词、搜索建议、统一搜索结果页
- 业务域导航与业务域入口编排
- 轮播、推荐位、卡片布局
- 文章详情页、案例详情页、词条详情页
- 同域最新、相关推荐、同标签推荐
- 知识广场展示与入口组织
- 技术问答前台交互流程
- 深度问答、联网搜索、文件搜索等能力入口编排
- 问答中的知识范围切换
- 应用市场展示
- 分享面板与分享前台体验
- 最近会话、复制、收藏等用户交互

一句话：

- 门户层负责怎么组织、怎么展示、怎么搜索、怎么问

### 6.2 知识空间层负责什么

如果 BiSheng 的知识空间只作为数据源，更适合只暴露数据原子能力：

- 空间列表、空间详情、描述、条目数、更新时间
- 文件夹树、文件列表、分页、排序
- 文件标题、摘要、业务域、内容类型、来源、发布时间、标签
- 文件正文、分段、附件、预览地址、原文件地址
- 标签列表、自定义元数据字段
- 同域、同标签、同空间、最近更新等推荐原料
- 基于空间/文件/文件夹的检索与对话能力
- 分享链接生成与分享信息查询

补充说明：

- 当前真实空间主模型是 `knowledge`
- 当前真实内容主模型是 `knowledge_file`
- 空间 `icon` 字段当前不适合作为门户首页稳定依赖字段

一句话：

- 知识空间层负责有什么数据、怎么取到

### 6.3 知识空间层不应承担什么

- 首页模块编排
- 轮播和推荐位编排
- 面向门户前端直接提供统一搜索接口
- 搜索结果页聚合展示
- 问答页会话体验与前台交互
- 应用市场展示
- 分享面板前台逻辑

## 7. 搜索边界

搜索能力明确归属门户层。

推荐模式：

1. 门户前端只调用门户自己的搜索接口
2. 门户后端统一做搜索索引、召回、聚合和结果编排
3. 门户后端内部再决定是否从知识空间数据源取数

不推荐模式：

1. 门户前端直接调用知识空间搜索接口
2. 前端自行拼装搜索结果、分页、筛选和混排

一句话：

- 搜索页面属于门户
- 搜索接口也应属于门户
- 知识空间只作为搜索数据来源之一

## 8. 当前 API 设计方向

### 8.1 直接不用做

当前已经明确不需要单独后端 API 的内容：

- 顶部导航
- 焦点展示区
- 应用市场
- 页脚
- 首页专属技术问答
- 权限相关
- 统计/热度相关

### 8.2 当前需要做

当前只需要收敛两类查询能力：

- 全局搜索接口
- 统一内容列表接口

### 8.3 门户统一搜索

建议接口：

- `GET /api/v1/portal/search`

用途：

- 服务首页全局搜索区
- 服务门户搜索结果页

建议输入：

- `q`
- `page`
- `page_size`
- `space_id`（可选）

返回原则：

- 返回统一内容结果列表
- 支持分页
- 支持按知识空间限定搜索范围

### 8.4 统一内容列表

建议接口：

- `GET /api/v1/portal/content-list`

用途：

- 服务业务域入口区
- 服务推荐内容区
- 服务专题内容区
- 服务知识广场区

建议输入：

- `space_id`
- `page`
- `page_size`
- `domain`（回显上下文）
- `query_type`（回显上下文）

返回原则：

- 前端决定查什么分类
- 前端决定怎么展示
- 后端只负责按条件返回结果列表

### 8.5 当前不纳入范围的接口

以下接口当前不在首页第一版确认范围内：

- `GET /api/v1/portal/content/{id}`
- `GET /api/v1/portal/content/{id}/related`
- `GET /api/v1/portal/spaces`
- `GET /api/v1/portal/spaces/{id}`
- `GET /api/v1/portal/spaces/{id}/children`
- `POST /api/v1/portal/qa/chat`
- `POST /api/v1/portal/share-link`

原因：

- 先收敛首页真正需要的结果查询能力
- 详情页、分享、问答、空间浏览后续按页面单独拆
- 避免先做过大的门户聚合层

### 8.6 搜索建议

当前保留为待确认项。

可选方案：

- 方案 A：第一版不做搜索建议
- 方案 B：如果后续需要提示词或联想词，不做 AI 推荐，只基于标签列表查询

当前推荐：

- 第一版先不做搜索建议接口
- 不做 AI 推荐型搜索建议
- 如果后续需要提示能力，直接基于标签列表查询

## 9. 当前推荐的最小实现方案

当前最克制的实现方式是：

- 只做 `GET /api/v1/portal/search`
- 只做 `GET /api/v1/portal/content-list`
- 搜索建议先不做

这样可以先支撑：

- 全局搜索
- 业务域内容返回
- 推荐内容返回
- 专题栏目内容返回
- 知识广场内容返回

## 10. 当前 BiSheng 可复用能力

以下能力在仓库中已经存在，可以优先复用：

- 知识空间 CRUD、成员、目录、搜索、标签、文件/文件夹对话
  - `src/backend/bisheng/knowledge/api/endpoints/knowledge_space.py`
- 传统知识库、文件、QA、预览、元数据
  - `src/backend/bisheng/knowledge/api/endpoints/knowledge.py`
  - `src/backend/bisheng/knowledge/api/endpoints/qa.py`
- 分享链接
  - `src/backend/bisheng/share_link/api/endpoints/share_link.py`
- Assistant OpenAPI 对话
  - `src/backend/bisheng/open_endpoints/api/endpoints/assistant.py`
- 全局 API 挂载
  - `src/backend/bisheng/api/router.py`

已知限制：

- 当前通用分享能力的资源类型枚举还没有覆盖知识空间和知识文件
- 门户统一搜索和统一内容列表还没有正式落代码

## 11. 后续建议

建议按以下顺序推进：

1. 单独定义门户统一内容 schema
2. 明确 schema 各字段的数据来源映射
3. 再定义 `portal/search` 和 `portal/content-list` 的最终输入输出
4. 最后决定前端哪些页面先接真实数据

当前建议优先补齐的内容字段：

- `summary`
- `source`
- `updated_at`
- `tags`

这些字段优先基于现有 `KnowledgeFile`、`Knowledge` 和标签体系承载，先避免大改核心表结构。

## 12. 内容查询设计

本节只讨论“内容怎么查”，不直接展开接口实现细节。

### 12.1 查询对象

门户内容查询的主对象先定为：

- `KnowledgeFile`

原因：

- 门户真正展示的是内容项
- 当前知识空间里的文件对象最接近门户内容
- `KnowledgeFile` 已具备标题、摘要、更新时间
- `KnowledgeFile` 能通过 `knowledge_id` 关联来源知识空间
- `KnowledgeFile` 能通过标签体系关联标签

### 12.2 前端分类与后端查询的关系

当前已确认的设计是：

- 前端固定分类
- 前端在 `domain` 上维护分类与知识空间的映射关系
- 推荐区域、专题区域等模块也都在前端完成配置和绑定
- 用户发起内容查询时，前端把映射后的 `space_id` 传给后端
- 后端直接根据 `space_id` 返回该知识空间下的具体文档内容

这意味着：

- `domain` 是前端分类概念
- `domain` 不是内容原生字段
- 后端不负责 `domain -> knowledge space` 的映射
- 后端不负责推荐区域、专题区域或其它 Space 功能入口的配置和绑定
- 后端真正执行查询时，核心筛选条件不是 `domain`，而是 `space_id`

一句话：

- 前端按分类组织
- 前端用 `domain` 做映射
- 前端手动配置推荐、专题及其它 Space 入口/功能绑定
- 后端按 `space_id` 查内容

### 12.3 第一版内容查询上下文

对于 `content-list`，第一版先只关注以下查询上下文：

- 分类标识
  - 例如前端自己的业务域 key、推荐 key、专题 key
  - 该字段用于标识“这是哪一类列表”
- 知识空间 ID
  - 这是后端真正执行内容筛选的核心条件
- 分页参数
  - `page`
  - `page_size`

其中：

- 分类标识主要用于前端上下文和响应标识
- 知识空间 ID 才决定后端查哪个知识空间下的内容

### 12.4 第一版实际筛选条件

第一版内容查询建议基于以下条件筛选 `KnowledgeFile`：

1. `knowledge_id = space_id`
   - 只查询当前指定知识空间中的内容
2. `file_type = FILE`
   - 不把文件夹当作门户内容项
3. 文件状态为可用内容
   - 优先只返回成功状态文件
4. 按更新时间倒序
   - 让门户先展示最近更新的内容

### 12.5 第一版查询链路

第一版建议查询链路如下：

1. 前端确定分类
2. 前端在 `domain` 上完成分类与知识空间的映射
3. 前端把映射后的 `space_id` 传给后端
4. 后端按 `space_id` 查询 `KnowledgeFile`
5. 后端补充来源知识空间名称和标签
6. 后端组装成 `PortalContentItem` 列表返回

### 12.6 当前不做的查询维度

第一版先不依赖以下维度作为稳定筛选条件：

- `content_type`
  - 当前没有稳定来源
- 内容原生 `domain`
  - 当前不是内容字段，而是前端分类概念
- 发布时间
  - 当前统一改为使用更新时间

### 12.7 对 API 的直接影响

基于当前内容查询设计，`content-list` 第一版更接近下面这类语义：

- 前端告诉后端：我要查哪个知识空间里的内容
- 后端返回这个知识空间中的文档列表

因此第一版 `content-list` 的最关键入参不是内容类型，也不是内容业务域，而是：

- `space_id`

分类字段可以保留，但它的定位更像：

- 上下文字段
- 前端分类标识
- 前端映射依据
- 响应回显字段

而不是后端内容主筛选条件

## 13. 模块覆盖设计

本节用于回答：当前这套设计已经能覆盖哪些具体功能模块，以及每个模块应该落到哪条查询链路上。

### 13.1 模块覆盖总原则

当前第一版优先覆盖这类模块：

- 模块的核心输出是“知识空间中的文档内容列表”
- 前端可以先通过 `domain` 或其它分类概念完成映射
- 后端最终只需要按 `space_id` 返回内容

对于这类模块，优先复用：

- `GET /api/v1/portal/content-list`

如果模块的核心能力是关键词检索，则优先复用：

- `GET /api/v1/portal/search`

### 13.2 当前模块覆盖矩阵

| 模块 | 当前是否覆盖 | 推荐查询方式 | 说明 |
| --- | --- | --- | --- |
| 首页业务域入口区 | 是 | `content-list` | 前端先做 `domain -> space_id` 映射，后端按 `space_id` 返回文档内容 |
| 首页推荐内容区 | 是 | `content-list` | 推荐区在前端完成配置和知识空间绑定，后端只按 `space_id` 返回文档列表 |
| 首页专题内容区 | 是 | `content-list` | 专题区在前端完成配置和知识空间绑定，后端只按 `space_id` 返回文档列表 |
| 首页知识广场中的内容列表 | 是 | `content-list` | 如果展示的是某知识空间下的文档内容，则前端绑定知识空间，后端复用同一查询链路 |
| 全局搜索区 | 是 | `search` | 关键词搜索文档内容，必要时可附带 `space_id` 限定范围 |
| 搜索结果页 | 是 | `search` | 与首页搜索区复用同一接口 |
| 知识详情页主体内容 | 是 | `spaces/{space_id}/files/{file_id}` | 详情对象是知识空间内文件，不需要父级路径 |
| 知识广场中的空间卡片 | 否 | 前端静态配置 | 空间卡片对象由前端写死配置，不需要后端接口 |
| 技术问答页 | 否 | BiSheng 日常模式 API | 问答范围由前端手动选择和配置，不纳入 `content-list` / `search` |
| 应用市场区 | 否 | 直接链接 | 前端直接写入并绑定 BiSheng Workflow 的免登录 URL，点击后直接跳转 |
| 相关推荐 / 同标签推荐 | 是 | `spaces/{space_id}/files/{file_id}/related` | 第一版不做复杂推荐，直接用当前文件第一个标签查询最新 5 条 |

### 13.3 已覆盖模块的统一模式

当前已经覆盖的模块，本质上都可以归为两类：

1. 内容列表类模块
   - 业务域入口区
   - 推荐内容区
   - 专题内容区
   - 知识广场中的内容列表

   统一模式：
   - 前端先确定分类、栏目或页面模块
   - 前端完成配置、绑定和到 `space_id` 的映射
   - 后端按 `space_id` 返回该知识空间下的文档内容

2. 搜索类模块
   - 首页全局搜索
   - 搜索结果页

   统一模式：
   - 前端传关键词
   - 如有需要，前端传 `space_id`
   - 后端按关键词搜索文档内容并返回统一结果项

### 13.4 当前第一版不覆盖的模块

以下模块当前不应该硬塞进 `content-list` 或 `search`：

- 知识空间卡片本身
  - 这是空间对象，不是文档内容对象
  - 当前由前端写死配置，不需要后端接口
- 技术问答页
  - 这是对话能力，不是内容列表能力
  - 直接复用 BiSheng 日常模式 API
  - 知识库范围由前端手动选择和配置
  - 前端筛选框不依赖后端返回额外类别字段
  - 集团知识库筛选范围不包含个人知识库内容
- 知识详情页接口
  - 已收敛为知识空间内文件详情接口
  - 下载能力由前端基于标识拼接链接
  - 不直接返回原始存储 URL
  - 父级路径不需要
- 分享能力
  - 需要单独确定分享资源类型和分享链接规则

### 13.5 第一版能支撑的功能边界

基于当前设计，第一版已经能稳定支撑：

- “给定某个知识空间，返回该空间下的文档内容列表”
- “给定关键词，返回文档内容搜索结果”
- “同一套内容结果项 schema 复用到多个门户列表模块”
- “给定 `space_id + file_id`，返回知识空间内文件详情”
- “给定 `space_id + file_id`，返回基于第一个标签的最新 5 条相关推荐”

第一版还不能完整支撑：

- 知识空间对象展示
- 相关推荐 / 同域推荐 / 同标签推荐
- 问答页交互能力

## 14. 第一版相关推荐设计

本节只定义第一版最简单可落地的相关推荐能力。

### 14.1 目标

- 不做复杂推荐系统
- 不做权重、召回、排序策略设计
- 只给知识详情页提供一个可用的相关推荐列表

### 14.2 推荐规则

第一版统一使用以下规则：

1. 读取当前文件的标签列表
2. 取第一个标签作为推荐标签
3. 查询带有同一标签的其他文件
4. 按 `update_time` 倒序取最新 5 条
5. 排除当前文件本身

### 14.3 兜底规则

- 如果当前文件没有标签，返回空列表
- 如果同标签内容不足 5 条，按实际命中数量返回

### 14.4 范围约束

第一版建议相关推荐的查询范围为：

- 当前用户可访问的知识空间范围

这样与全局搜索范围保持一致，不额外引入新的权限或推荐边界设计。

## 15. 权限设计

### 15.1 第一版权限策略

第一版不做权限校验，门户内容完全公开，匿名可访问。

具体做法：

1. 接口层：当前不加鉴权依赖，匿名可访问
2. Service 层：查询方法预留 `user_id: Optional[int] = None` 参数，当前不传就不过滤
3. 查询逻辑：当前只查 `is_released=True` / `auth_type=public` 的空间内容，天然就是公开数据
4. 后续扩展：需要权限时，加一个鉴权中间件 + 在 Service 层根据 `user_id` 过滤可访问的 `space_id` 即可

### 15.2 设计理由

- 门户场景零摩擦，不用登录就能用
- 其他场景复用时，只需注入鉴权依赖，Service 层逻辑不用改
- 查询条件本身就限定了公开数据，不存在泄露风险
