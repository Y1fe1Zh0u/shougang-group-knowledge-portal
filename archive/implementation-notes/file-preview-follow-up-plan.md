# 文件预览后续方案

## 背景

当前门户详情页已经修复了“把下载地址直接塞进 iframe 导致触发下载”的问题。

现阶段的行为是：

- 优先使用 BiSheng 返回的 `preview_url`
- 如果没有可直接渲染的预览地址，则不再把 `original_url` 当作 iframe 预览源
- 对无可渲染预览地址的文件，门户改为读取已解析的 chunk 内容并直接渲染正文文本

这解决了“用户点开详情页只会下载文件”的问题，但还没有达到 BiSheng 那套按文件类型原样预览的体验。

## 已确认事实

### 1. `preview_url` 不是必然存在

对当前线上样本文件，BiSheng 预览接口返回：

- `original_url` 有值
- `preview_url` 为空字符串

同时文件元数据中的 `preview_file_object_name` 也为空。

结论：

- 正常上传后的文件，并不保证一定生成持久化 `preview_url`
- “能在 BiSheng 页面里看见内容”不等于“文件记录里一定有 `preview_url`”

### 2. BiSheng 有多套“预览”逻辑

BiSheng 前端并不是只靠 `preview_url + iframe`。

实际存在至少三类预览来源：

1. `preview_url`
   - 若后端已有预览资源，前端优先使用
2. `original_url`
   - 某些文件类型没有 `preview_url` 时，前端会直接拉原文件做渲染
3. 预览任务产物
   - `POST /api/v1/knowledge/preview`
   - 再轮询 `/api/v1/knowledge/preview/status`
   - 最终拿到任务结果中的 `file_url`

### 3. BiSheng 是按文件类型分别渲染的

已确认的前端策略：

- `pdf`：专用 PDF viewer
- `docx`：下载原文件后，在浏览器里转换为 HTML 再显示
- `xlsx/xls/csv`：浏览器端解析表格并渲染
- `txt/md/html`：文本类 viewer
- `png/jpg/...`：直接图片预览
- `ppt/pptx`：当前直接显示“不支持预览”

因此，门户如果要真正“和 BiSheng 一样”，需要复刻的是：

- 预览源选择逻辑
- 文件类型对应渲染器

而不是单纯沿用一个 iframe。

## 当前门户状态

当前门户已经具备：

- 预览/下载 URL 归一化
- 避免把 `original_url` 当 iframe 预览源
- 新增 chunks 接口：
  - `GET /api/v1/knowledge/space/{space_id}/files/{file_id}/chunks`
- 当没有可渲染预览地址时，详情页直接渲染解析后的 chunk 正文

当前门户仍缺少：

- PDF 原样预览
- DOCX 原样预览
- Excel/CSV 原样预览
- 统一预览源编排层
- 预览任务结果缓存/复用策略

## 后续目标

目标不是继续“避免下载”，而是逐步达到以下体验：

- 用户进入详情页后，能在页面内直接看到可读内容
- 对主流格式，尽量接近 BiSheng 的原样预览体验
- 不把实现细节暴露给用户
- 下载按钮始终只承担下载语义，不再参与预览语义

## 建议实施顺序

### 第一阶段：PDF 正式预览

目标：

- 对 `pdf` 接入专用 PDF viewer
- 不再依赖简单 iframe

原因：

- 收益最高
- 技术路径清晰
- 与 chunk 高亮、定位扩展性最好

建议：

- 优先采用与 BiSheng 接近的同域代理 URL 方案
- 统一处理 worker、分页、加载失败 fallback

### 第二阶段：DOCX 预览

目标：

- 对 `docx` 在浏览器中拉取文件并转换为 HTML 显示

原因：

- 业务文档里 DOCX 很常见
- 仅靠下载体验不够

注意：

- `.doc` 与 `.docx` 分开处理
- `.doc` 可以先继续显示“不支持在线预览”

### 第三阶段：文本类与图片类整理

目标：

- `txt/md/html/image` 统一为正式预览组件

内容：

- `txt`：纯文本 viewer
- `md`：Markdown viewer
- `html`：受控渲染
- 图片：统一缩放、滚动、背景处理

### 第四阶段：表格类预览

目标：

- `xlsx/xls/csv` 做浏览器端表格解析与展示

注意：

- 这一阶段复杂度明显上升
- 需要关注大文件性能、图片单元格、编码兼容等问题

### 第五阶段：预览任务产物编排

目标：

- 不再让详情页自己猜该用哪个 URL
- 在 BFF 层统一给前端返回“可渲染预览源”

建议：

- 增加一个门户内部的“预览源决策层”
- 输入：
  - `preview_url`
  - `original_url`
  - 文件后缀
  - 是否已有 chunk 内容
  - 是否需要触发预览任务
- 输出：
  - 预览模式
  - 渲染地址
  - 降级策略

## 推荐技术方向

### 方向 A：渐进增强

策略：

- 先保留当前 chunks fallback
- 分文件类型逐步替换为正式 viewer

优点：

- 风险最低
- 可逐步上线

缺点：

- 一段时间内体验不完全一致

### 方向 B：一次性接近 BiSheng

策略：

- 同时补 PDF、DOCX、文本、表格等 viewer

优点：

- 一次完成体验统一

缺点：

- 依赖与实现复杂度明显上升
- 验证面更大

建议选择：

- 优先采用方向 A

## 验收标准

### 最低标准

- 不再因为预览逻辑触发浏览器下载
- 无 `preview_url` 时，详情页仍可展示正文内容

### 中期标准

- PDF 能在页内稳定预览
- DOCX 能在页内稳定预览
- 下载按钮始终保留

### 完整标准

- PDF / DOCX / XLSX / CSV / TXT / MD / HTML / 图片均有稳定预览
- `ppt/pptx/doc` 明确展示不支持
- 预览逻辑在 BFF 和前端中都有清晰边界

## 主要风险

### 1. 鉴权与同域访问

BiSheng 自己的 viewer 会把远端 URL 替换成同域路径后再 fetch。

门户若直接拉远端签名地址，可能遇到：

- 浏览器下载行为
- 跨域问题
- 预览组件无法稳定读取

### 2. 大文件性能

DOCX、XLSX 在浏览器端解析时需要控制：

- 首屏加载时长
- 内存占用
- 卡顿

### 3. 文件类型差异

并不是所有后缀都适合同一类预览方式。

例如：

- `pdf` 更适合专用 viewer
- `docx` 更适合转换后显示
- `ppt/pptx` 当前应保持不支持

## 结论

当前修复解决的是“不能预览、只会下载”的错误行为。

后续真正要做的是：

- 从“能展示内容”升级到“按文件类型原样预览”
- 把预览能力从单一 iframe 思路升级为“预览源选择 + 文件类型渲染器”

建议从 `pdf + docx` 开始做下一轮实现。
