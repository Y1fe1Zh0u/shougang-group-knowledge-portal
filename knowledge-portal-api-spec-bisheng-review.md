- 把原模块设计里的非 API 内容完整搬到同一份文档

- 推荐能力在服务层做封装，后续可以切换推荐系统而不改对外 API

- 模块说明、UI 线框、首页设计、页面清单、后台配置等内容都以本文为准

## 3. 接口清单

### 3.1 与之前文档的对照结论

之前文档中的“新增接口 / 复用现有接口”划分比较粗，容易把下面两种情况混在一起：

- 直接复用 BiSheng 已存在的同名同方法接口
- 复用 BiSheng 已有查询能力，但对外封装成门户自己的 API

因此本稿改成以下三档：

1. 直接复用现有 API
2. 复用现有后端能力后封装成门户 API
3. 门户新增能力或新增服务封装

### 3.2 直接复用现有 API

| 接口 | 路径 | 入参 | 出参 | 说明 |
| --- | --- | --- | --- | --- |
| 空间标签列表 | `GET /api/v1/knowledge/space/{space_id}/tag` | `space_id` | `UnifiedResponseModel<Tag[]>` | BiSheng 已有，直接复用 |
| 文件预览地址 | `GET /api/v1/knowledge/space/{space_id}/files/{file_id}/preview` | `space_id`、`file_id` | `UnifiedResponseModel<FilePreviewData>` | BiSheng 已有，直接复用 |
| 技术问答 / AI Overview | `POST /api/v1/workstation/chat/completions` | `text`、`clientTimestamp`、`model`、`use_knowledge_base`、`files` 等 | `SSE` 流式事件 | BiSheng 日常模式 API，直接复用 |

### 3.3 复用现有后端能力后封装成门户 API

| 门户接口 | 路径 | 入参 | 出参 | 复用基础能力 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 单空间文件列表 | `GET /api/v1/knowledge/space/{space_id}/files` | `space_id`、`file_ext`、`tag`、`page`、`page_size` | `UnifiedResponseModel<PagedKnowledgeFileData>` | `GET /api/v1/knowledge/space/{space_id}/children`、`GET /api/v1/knowledge/space/{space_id}/search` | 对单空间文件查询能力做门户化封装，补充更适合前端的筛选和返回结构 |
| 跨空间文件查询 | `GET /api/v1/knowledge/files` | `q`、`tag`、`space_ids`、`file_ext`、`sort`、`page`、`page_size` | `UnifiedResponseModel<PagedKnowledgeFileData>` | 单空间搜索、标签体系、SQL 元数据查询、ES 检索 | 将跨空间全文搜索和按标签聚合统一收口为门户查询接口 |
| 文件详情 | `GET /api/v1/knowledge/space/{space_id}/files/{file_id}` | `space_id`、`file_id` | `UnifiedResponseModel<KnowledgeFileDetail>` | `KnowledgeFile` / `Knowledge` / `Tag` 元数据查询能力 | 对文件基础元信息做门户详情封装 |

说明：

- 这些接口不一定在 BiSheng 中以相同 `method + path` 已存在
- 但底层依赖的查询链路、数据模型和服务能力大部分已经存在
- 因此更准确的说法是“复用现有能力封装成门户 API”，而不是完全从零新做

### 3.4 门户新增能力或新增服务封装

| 门户接口 / 能力 | 路径 | 入参 | 出参 | 说明 |
| --- | --- | --- | --- | --- |
| 相关推荐 | `GET /api/v1/knowledge/space/{space_id}/files/{file_id}/related` | `space_id`、`file_id`、`limit` | `UnifiedResponseModel<RelatedKnowledgeFileData>` | 对外是稳定门户 API，内部通过 `RecommendationService` 封装，第一版可用现有标签和更新时间规则实现，后续可切换推荐系统 |

## 4. 公共数据结构

### 4.1 `KnowledgeFileItem`（列表项）

用于单空间列表、跨空间查询、相关推荐等所有列表场景。

```json
{
  "id": 1580,
  "space_id": 12,
  "title": "热轧1580产线精轧机振动纹治理实践",
  "summary": "针对 1580 热连轧精轧机组出现的周期性振动纹缺陷...",
  "source": "轧线技术案例库",
  "updated_at": "2026-04-13T10:30:00",
  "tags": ["热轧", "精轧机", "振动纹"]
}
```

| 字段 | 类型 | 必填 | 来源 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | `int` | 是 | `KnowledgeFile.id` | 文件 ID |
| `space_id` | `int` | 是 | `KnowledgeFile.knowledge_id` | 所属知识空间 ID，用于详情 / 预览 / 推荐接口串联 |
| `title` | `string` | 是 | `KnowledgeFile.file_name` | 去掉文件后缀后的标题 |
| `summary` | `string` | 是 | `KnowledgeFile.abstract` | 摘要，空值返回 `""` |
| `source` | `string` | 是 | `Knowledge.name` | 所属知识空间名称 |
| `updated_at` | `string` | 是 | `KnowledgeFile.update_time` | 更新时间 |
| `tags` | `string[]` | 是 | `Tag.name` | 标签名称数组，空时返回 `[]` |

### 4.2 `KnowledgeFileDetail`（详情）

在 `KnowledgeFileItem` 基础上增加：

```json
{
  "id": 1580,
  "space_id": 12,
  "title": "热轧1580产线精轧机振动纹治理实践",
  "summary": "针对 1580 热连轧精轧机组...",
  "source": "轧线技术案例库",
  "updated_at": "2026-04-13T10:30:00",
  "tags": ["热轧", "精轧机", "振动纹"],
  "space": {
    "id": 12,
    "name": "轧线技术案例库"
  }
}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `space` | `{id: int, name: string}` | 是 | 所属知识空间信息 |

说明：

- 第一版详情接口不直接返回正文内容
- 正文通过 `preview_url` 渲染展示

### 4.3 `PagedKnowledgeFileData`

## 5. 通用约定

### 5.1 路由风格

- 全局前缀统一为 `/api/v1`
- 知识相关接口挂在 `/knowledge` 或 `/knowledge/space`
- 问答相关接口继续复用 `/workstation/chat/completions`

### 5.2 响应包装

普通 HTTP 接口统一采用 BiSheng 现有响应包装：

```json
{
  "status_code": 200,
  "status_message": "SUCCESS",
  "data": {}
}
```

说明：

- 不使用 `{code, message, data}`
- 成功响应默认通过 `resp_200(data=...)` 返回
- SSE 接口不套该包装，保持流式输出

### 5.3 分页约定

- `page` 默认 `1`
- `page_size` 默认 `20`
- `page_size` 最大 `100`

### 5.4 查询范围约定

- 第一版默认只查询门户后台已绑定且公开发布的知识空间
- 跨空间查询接口允许通过 `space_ids` 显式传入白名单范围
- `space_ids` 类型为 `int[]`
- 推荐场景默认也限定在门户可见空间范围内
- 业务域只是前端展示和后台配置概念，不作为后端查询核心字段
- 后端查询只认 `space_id` 或 `space_ids`

### 5.5 类型约定

- `id`、`space_id`、`file_id` 均使用 `int`
- 时间字段使用后端序列化后的字符串
- 文件格式筛选统一使用 `file_ext`
- 不使用 `file_type` 表示文件后缀，避免和数据库里的文件 / 文件夹类型混淆
- `domain` 不进入 API 入参、出参和内容结果项

### 5.6 权限约定

- 第一版不新增鉴权逻辑
- 只读取公开数据
- 后续若需要扩权，可在服务层按 `user_id` 过滤可见空间范围

## 6. 模块一：全局搜索

### 6.1 功能描述

首页顶部搜索框，用户输入关键词，跨门户可见空间搜索文件内容。

### 6.2 前端职责

- 搜索框 UI 和交互
- 热门搜索词展示
- 搜索结果跳转到搜索结果页

### 6.3 后端职责

- 提供统一跨空间查询接口
- 在 `q` 存在时执行标题、摘要、正文搜索
- 返回统一的 `KnowledgeFileItem`

### 6.4 接口

#### 6.4.1 请求方式

```text
GET /api/v1/knowledge/files?q=振动纹&space_ids=12&space_ids=18&page=1&page_size=20
```

#### 6.4.2 入参

| 参数 | 位置 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| `q` | query | `string` | 是 | 搜索关键词 |
| `space_ids` | query | `int[]` | 否 | 查询范围白名单，推荐使用多值 query 形式 |
| `page` | query | `int` | 否 | 默认 `1` |
| `page_size` | query | `int` | 否 | 默认 `20`，最大 `100` |

#### 6.4.3 出参

最外层响应：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `status_code` | `int` | 响应状态码 |
| `status_message` | `string` | 响应消息 |
| `data` | `PagedKnowledgeFileData` | 分页结果 |

响应示例：

```json
{
  "status_code": 200,
  "status_message": "SUCCESS",
  "data": {
    "data": [
      {
        "id": 1580,
        "space_id": 12,
        "title": "热轧1580产线精轧机振动纹治理实践",
        "summary": "针对 1580 热连轧精轧机组出现的周期性振动纹缺陷...",
        "source": "轧线技术案例库",
        "updated_at": "2026-04-13T10:30:00",
        "tags": ["热轧", "精轧机", "振动纹"]
      }
    ],
    "total": 23,
    "page": 1,
    "page_size": 20
  }
}
```

### 6.5 搜索实现方案

1. API 入口统一为 `GET /api/v1/knowledge/files`
2. `q` 存在时走全文检索链路
3. 默认查询范围为门户后台已绑定且公开发布的空间
4. 如果传了 `space_ids`，则进一步按该白名单范围过滤
5. ES 负责正文检索
6. SQL 负责元数据补全和结果筛选
7. 最终组装为 `KnowledgeFileItem`

## 7. 模块二：搜索结果页

### 7.1 功能描述

展示搜索结果列表，支持筛选、排序、分页。顶部展示 AI Overview。

### 7.2 前端职责

- AI Overview 区域的 SSE 流式渲染
- 业务域、文档类型、标签筛选
- 排序切换
- 搜索结果列表渲染
- 分页控件
- 点击结果跳转到详情页

### 7.3 后端职责

- 复用统一跨空间查询接口
- 复用现有日常模式 API

### 7.4 接口

前端搜索时并行调用：

1. `GET /api/v1/knowledge/files`
2. `POST /api/v1/workstation/chat/completions`

#### 7.4.1 跨空间文件查询

请求方式：

```text
GET /api/v1/knowledge/files?q=振动纹&tag=热轧&file_ext=pdf&space_ids=12&sort=relevance&page=1&page_size=20
```

入参：

| 参数 | 位置 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| `q` | query | `string` | 否 | 全文搜索关键词 |
| `tag` | query | `string` | 否 | 标签名称筛选 |
| `file_ext` | query | `string` | 否 | 文件后缀筛选 |
| `space_ids` | query | `int[]` | 否 | 查询范围白名单 |
| `sort` | query | `string` | 否 | `relevance` 或 `updated_at` |
| `page` | query | `int` | 否 | 默认 `1` |
| `page_size` | query | `int` | 否 | 默认 `20`，最大 `100` |

规则：

1. `q` 和 `tag` 不能同时为空
2. `q` 存在且未显式传 `sort` 时，默认 `relevance`
3. 仅 `tag` 查询且未显式传 `sort` 时，默认 `updated_at`
4. `sort=updated_at` 时按更新时间倒序

出参：

- 最外层：`UnifiedResponseModel`
- `data`：`PagedKnowledgeFileData`

响应示例：

```json
{
  "status_code": 200,
  "status_message": "SUCCESS",
  "data": {
    "data": [
      {
        "id": 1580,
        "space_id": 12,
        "title": "热轧1580产线精轧机振动纹治理实践",
        "summary": "针对 1580 热连轧精轧机组出现的周期性振动纹缺陷...",

      }
    ],
    "total": 23,
    "page": 1,
    "page_size": 20
  }
}
```

## 8. 模块三：业务域导航

### 8.1 功能描述

首页展示多个业务域，点击进入对应业务域的文件列表页，支持按文件格式和标签筛选。

### 8.2 前端职责

- 业务域卡片 / 图标展示

- 从后台配置读取业务域与 `space_id` 的映射关系
- 进入业务域页面后并行调用文件列表和标签列表
- 筛选交互

### 8.3 后端职责

- 按 `space_id` 返回文件列表
- 支持 `file_ext` 和 `tag` 筛选
- 返回空间下全部标签

### 8.4 接口

#### 8.4.1 文件列表

请求方式：

```text
GET /api/v1/knowledge/space/{space_id}/files?file_ext=pdf&tag=热轧&page=1&page_size=20
```

入参：

| 参数 | 位置 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| `space_id` | path | `int` | 是 | 知识空间 ID |
| `file_ext` | query | `string` | 否 | 文件格式筛选 |
| `tag` | query | `string` | 否 | 标签筛选 |
| `page` | query | `int` | 否 | 默认 `1` |
| `page_size` | query | `int` | 否 | 默认 `20`，最大 `100` |

出参：

- 最外层：`UnifiedResponseModel`

```json

  "data": {
```

#### 8.4.4 前端路由约定

建议区分“业务域入口语义”和“知识空间查询实现”：

- 业务域聚合页：`/domains`
- 业务域详情页：`/domain/{domain_name}`
- 知识空间页：`/space/{space_id}`

说明：

- 用户从首页业务域卡片或业务域聚合页进入时，优先走 `/domain/{domain_name}`
- 前端再根据后台配置把 `domain_name` 解析到一个或多个 `space_id`
- 这样后续即使业务域从“一对一绑定空间”变成“一对多绑定空间”，前端入口和页面标题也不需要改路由语义

#### 8.4.5 当前顺序逻辑（前端现状）

当前前端实现中，业务域顺序直接取前端 / 后台配置数组顺序。

具体规则：

1. 首页业务域导航按业务域配置数组顺序展示
2. 全部业务域页按同一份业务域配置数组顺序展示
3. 首页业务域导航当前只取业务域配置数组前 `N` 个展示
4. `N` 由前端展示配置控制

因此当前顺序逻辑可以理解为：

- `CFG.domains` / 后台业务域配置列表的声明顺序 = 全部业务域页顺序
- 首页顺序 = `CFG.domains.slice(0, N)` 的顺序
- 当前不单独维护 `show_on_home` / `home_order`

后续建议：

- 当前阶段继续保持轻逻辑即可
- 如果后续接后端并需要更强的后台可编辑能力，再补显式配置字段

#### 9.5.2 推荐能力封装

统一服务上下文建议：

```text
scene: portal_section | file_related
space_ids: int[]
anchor_space_id: int | null
anchor_file_id: int | null
tag: string | null
limit: int
```

第一版默认 provider：

- `tag_feed`

推荐服务默认规则：

- `portal_section`：按 `tag + updated_at`
- `file_related`：按 `same_tag + updated_at`

后续可替换 provider：

- `vector_similarity`
- `hybrid_recall`
- `external_recsys`

切换原则：

- API 路径不变
- 入参不变
- 出参不变
- 仅在服务配置或后台配置中切换 provider

## 16. 门户后台配置管理

### 16.1 功能描述

门户需要一个后台配置页面，用于管理业务域、知识空间、首页分区、问答范围和推荐策略。

### 16.2 配置内容

| 配置项 | 说明 |
| --- | --- |
| 知识空间绑定 | 将 BiSheng 知识空间绑定到门户，形成门户可见范围 |
| 业务域（置顶板块） | 从已绑定空间中选择置顶展示的空间，配置业务域名称和图标 |
| 绑定的 `space_id` | 业务域与知识空间一对一绑定 |
| tags 缓存 | 绑定时自动缓存空间 tags，用于前端筛选 |
| 首页分区标签 | 配置推荐 / 专题分区对应的标签名 |
| 日常模式 `knowledge_space_ids` | 技术问答 / AI Overview 的知识空间范围 |
| 推荐策略配置 | 推荐服务使用的 provider / strategy |
| 应用市场配置 | 应用卡片名称、图标、描述、跳转 URL |

### 16.3 使用场景

- 业务域页面标签筛选下拉框使用对应空间的 tags 缓存
- 搜索结果页标签筛选下拉框使用所有已绑定空间的 tags 合集
- 搜索结果页业务域筛选使用后台配置的业务域列表
- 业务域页面文件列表使用绑定的 `space_id`
- 全局搜索默认范围使用后台绑定的 `space_ids`
- AI Overview 和技术问答使用后台配置的 `knowledge_space_ids`
- 推荐服务使用后台配置的 provider / strategy

## 17. 首页设计

### 17.1 首页模块布局（从上到下）

| 顺序 | 模块 | 数据来源 | 说明 |
| --- | --- | --- | --- |
| 1 | 顶部导航 + 全局搜索 | 前端写死 + 热门搜索词 | 搜索框 + 热门搜索标签 |
| 2 | Banner 轮播 | 前端写死 / 后台配置 | 内容待定 |
| 3 | 业务域导航 | 后台配置 + 单空间列表接口 | 首页按业务域配置数组顺序取前 N 个展示，并提供“全部业务域”入口 |
| 4 | 推荐 / 专题分区 | `GET /api/v1/knowledge/files?tag=...` | 按标签聚类，每个分区展示若干条卡片 |
| 5 | 知识广场 | 后台配置 + 文件数量统计 | 首页按知识空间配置数组顺序取前 N 个展示 |
| 6 | 技术问答入口 | 前端写死 | 热门问题 + 去提问按钮 |
| 7 | 应用市场 | 后台配置 | 应用卡片，点击跳转 Workflow URL |
| 8 | 页脚 | 前端写死 | 版权信息等 |

### 17.2 首页接口调用

- 各推荐 / 专题分区并行调用 `GET /api/v1/knowledge/files?tag=...&page_size=4`
- 知识广场文件数量通过批量统计服务获取
- 其他模块使用前端静态配置或后台配置

### 17.3 首页当前排序逻辑（前端现状）

当前首页两块入口区都采用轻量逻辑：

- 业务域导航：按业务域配置数组顺序取前 `N` 个
- 知识广场：按知识空间配置数组顺序取前 `N` 个

说明：

- 当前不单独维护首页展示专用排序字段
- 是否展示与排序都通过“数组位置 + `slice` 数量控制”完成
- 业务域和知识广场当前保持同一套前端控制思路

### 17.4 知识广场卡片

每张空间卡片展示：

- 空间名称
- 文件数量
- 点击进入该空间的通用列表页

### 17.5 技术问答入口

- 展示 3 到 5 个热门问题
- “去提问”按钮跳转到技术问答页

### 17.6 Banner 轮播

可选方向：

- 平台介绍 / 宣传文案
- 重点专题入口
- 后台配置维护图片和跳转链接
